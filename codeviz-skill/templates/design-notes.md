# CodeViz 导图视觉设计说明 - 墨白

> 这份文档解释每个设计决策的理由，方便吴八哥接后端数据时理解意图。

## 一、配色系统：不是四色块，是四态三层

现有版本的问题：只用 `border-left-color` 区分状态，颜色单一没层次。

**我的方案**：每个状态有 **填充色 + 边框色 + 文字色** 三层，外加 glow 光晕。

### 亮色模式
| 状态 | 填充 | 边框 | 文字 | 用途 |
|------|------|------|------|------|
| 未开始 | `#F4F4F5` | `#D4D4D8` | `#71717A` | 卡片底色淡灰 |
| 进行中 | `#FEF3C7` | `#F59E0B` | `#92400E` | 琥珀色，醒目但不刺眼 |
| 已完成 | `#D1FAE5` | `#10B981` | `#065F46` | 翡翠绿，降低不透明度弱化 |
| 报错 | `#FEE2E2` | `#EF4444` | `#991B1B` | 红色 + 光晕环，最强视觉警示 |

### 暗色模式（关键：不是简单反色）
暗色模式下填充色用**深色**（不是亮色的反色），边框用**亮色**，文字用**浅色**——保证对比度。

| 状态 | 填充 | 边框 | 文字 |
|------|------|------|------|
| 未开始 | `#2A2A30` | `#52525B` | `#A1A1AA` |
| 进行中 | `#422006` | `#F59E0B` | `#FCD34D` |
| 已完成 | `#064E3B` | `#10B981` | `#6EE7B7` |
| 报错 | `#7F1D1D` | `#EF4444` | `#FCA5A5` |

**为什么这样设计**：暗色背景上，如果填充色太亮会刺眼，太暗又看不出状态。用深填充+亮边框+浅文字的组合，既识别度高又不疲劳。

## 二、布局策略：Phase 分组 + 卡片网格

### 现有问题
平铺所有任务卡片，没有阶段感，看不出项目结构。

### 我的方案
1. **按 Phase 分组**：每个 Phase 是独立区块，有标题 + 进度条 + 完成数
2. **Phase 内卡片网格**：`auto-fill, minmax(300px, 1fr)`，自动响应屏幕宽度
3. **左侧栏固定 260px**：放项目列表 + 进度环 + 统计网格

### 为什么选卡片网格而不是树状连线
- **依赖关系太复杂时会乱**：T008 依赖 T003 和 T007，跨 Phase 连线会穿越其他卡片
- **卡片网格信息密度更高**：每个卡片能放文件路径、commit、错误信息
- **依赖关系用 chip 标签表达**：`依赖 T003 T004`，比连线更清晰
- 树状连线留 v2，作为可选视图

## 三、动效设计：每个都有具体曲线

### 1. 状态变化脉冲（核心动效）
任务状态改变时，卡片会有一次 `scale(1) → scale(1.03) → scale(1)` 的脉冲。
```css
animation: task-pulse 600ms cubic-bezier(0.4, 0, 0.6, 1);
```
**理由**：让用户余光就能感知到"有任务变了"，不用盯着看。

### 2. 进行中图标旋转
进行中任务的状态图标持续旋转（2秒一圈）。
**理由**：旋转 = 活跃，静态 = 停滞。最直观的"正在进行"暗示。

### 3. 报错抖动
报错任务出现时，图标左右抖动一次。
**理由**：抖动 = 异常，跟手机震动一个隐喻。

### 4. hover 磁吸
hover 时 `translateY(-3px)` + 阴影加深 + 左侧色条变宽。
```css
transition: all 250ms cubic-bezier(0.25, 0.46, 0.45, 0.94);
```
**理由**：鼠标靠近时卡片"抬起"，反馈明确但不夸张。

### 5. Phase 入场
每个 Phase 加载时从下方淡入，`animation-delay` 错开 60ms。
**理由**：避免一次性刷出来眼花，错落入场有节奏感。

### 6. 进度条/进度环过渡
宽度变化用 `600ms cubic-bezier(0.16, 1, 0.3, 1)`（spring 弹性曲线）。
**理由**：进度增长要有"弹性感"，不是机械的线性。

## 四、字体层级

| 元素 | 字号 | 字重 | 字体 |
|------|------|------|------|
| 项目标题 | 22px | 600 | 系统字体 |
| Phase 名 | 15px | 600 | 系统字体 |
| 任务名 | 14px | 500 | 系统字体 |
| 任务描述 | 12px | 400 | 系统字体 |
| 任务 ID | 10px | 500 | SF Mono |
| 文件路径 | 10px | 400 | SF Mono |
| commit hash | 10px | 400 | SF Mono |
| 统计数字 | 20px | 600 | 系统字体 |

**原则**：等宽字体（SF Mono）只用在 ID、文件路径、commit hash 上——这些是需要精确对齐的"代码类"信息。其他用系统字体，保证原生体验。

## 五、文件存在状态的视觉表达

任务关联的文件，存在和不存在视觉不同：
- **存在**（`.exists`）：绿色边框 + 绿色填充（暗示"有进展"）
- **不存在**：灰色边框 + 灰色填充

**需要后端提供**：`task.fileStats = { existing: 2, total: 3 }`，前端逐个文件判断。

## 七、性能考量

1. **50 个任务不卡**：所有动效只用 `transform` 和 `opacity`（GPU 加速），不触发布局重排
2. **脉冲动画用 class 触发**：不是常驻动画，状态变化时加 `.pulse` 类，600ms 后移除
3. **进行中旋转只转图标**：不转整个卡片，减少重绘范围
4. **暗色模式用 CSS 变量**：切换时不用重新渲染 DOM

---

## 八、响应式

- **宽屏（>900px）**：左侧栏 260px + 右侧导图区，双屏工作流
- **窄屏（≤900px）**：左侧栏收到顶部，导图区占满
